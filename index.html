Project Focus

Train a compact GPT-style Transformer on TinyStories under constrained compute, then evaluate whether the resulting model can generate coherent stories efficiently.

The project combines data preprocessing, mixed precision training, GPU-aware batching, validation perplexity tracking, and KV-cache-ready inference.

Introduction

Why This Project Matters

02 / 12

The Challenge

  • Large language models usually require compute budgets far beyond a class project.
  • Fast experimentation is difficult when training cost dominates every design choice.
  • Small models make it possible to study the full NLP pipeline rather than only a fragment.

Our Approach

  • Use TinyStories because it is language-rich, bounded, and tractable for small Transformers.
  • Train with AMP, tuned batching, checkpointing, and cached token tensors.
  • Measure validation loss and perplexity, then connect training quality to inference efficiency.

Problem Statement

Constrained Optimization for Language Modeling

03 / 12

Model Capacity

Depth, hidden size, and attention heads improve expressiveness, but larger models reduce the number of updates possible within a fixed training window.

Training Efficiency

Batch size, gradient accumulation, learning rate schedule, mixed precision, and data loading all shape convergence speed.

Generalization

Held-out validation perplexity is the main quantitative signal, so the model has to learn the distribution rather than memorize examples.

Experiment Design

Research Questions

04 / 12

RQ1 - Efficiency vs. Quality

How do inference settings, especially KV caching and sliding window cache size, affect latency, GPU memory, and generation quality in a small Transformer trained on TinyStories?

RQ2 - Context Reuse

Does greater reuse of previously computed context align with lower continuation perplexity on story-completion tasks?

Final Run Scope

The completed run establishes the trained baseline and validation curve. Full cache-window comparisons remain the next experimental extension.

Hypothesis

Efficient training can produce a capable story model, while cache-based inference should reduce repeated computation with a measurable memory-quality tradeoff.

What to Remember

  • A Transformer block repeatedly updates every token's vector.
  • Attention is the communication step: tokens decide which earlier tokens matter.
  • Multiple heads let the model track different relationships at the same time.
  • The final logits are a ranked list of candidate next tokens.
output = TransformerBlock(input_vectors)

Experiment Procedure

What Was Measured

07 / 12

Training Loop

  • Train a fixed GPT-style Transformer on cached TinyStories tensors.
  • Log training loss and perplexity every 100 steps.
  • Evaluate validation loss and perplexity every 500 steps.
  • Track learning rate decay and GPU memory use.

Evaluation Metrics

  • Validation cross-entropy loss and perplexity.
  • Training perplexity trend over the run.
  • Allocated, reserved, and peak allocated GPU memory.
  • Planned follow-up: latency, cache hit rate, continuation perplexity, and story quality scoring.

Conclusions & Future Scope

What the Final Run Shows

10 / 12

Conclusions

  • The cached preprocessing pipeline and memory-mapped dataset made repeated training practical.
  • The model reached validation perplexity 7.42, showing meaningful distribution learning on TinyStories.
  • GPU allocation stabilized after warmup, which supports reproducible follow-up inference experiments.

Future Scope

  • Complete full KV-cache vs. sliding-window cache comparisons at W = 64, 128, and 256.
  • Measure latency per token, cache hit rate, continuation perplexity, and story-quality scores.
  • Compare generated stories qualitatively and evaluate whether lower perplexity tracks coherence.
